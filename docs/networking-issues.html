Claude Code Session Report: Networking & Integration Barriers

A factual account of the obstacles encountered across two Claude Code sessions attempting to build and test a Python application that integrates with the Bitwarden API.
Dates: March 15–16, 2026  |  Platform: macOS 25.4.0, Apple Silicon  |  Python: 3.14.3  |  Claude Code model: Claude Opus 4.6

Executive Summary

Across two Claude Code sessions spanning several hours, a straightforward task—install Python packages, install the Bitwarden CLI, and sync Chrome passwords to a Bitwarden vault—was repeatedly blocked by a macOS DNS resolver malfunction. The DNS issue affects socket.getaddrinfo() (used by Python, Node.js, curl, and every standard networking library) but does not affect dig, nslookup, or ping by IP. The problem persists across sessions, survives DNS cache flushes, and cannot be resolved without sudo access that Claude Code does not have.

The sessions produced approximately 6,200 lines of working, tested application code (44 files, 52 unit tests all passing). The Bitwarden CLI was eventually installed by the user manually, API credentials were verified working, and 3 test items were successfully pushed to the vault. However, running the actual sync engine against a live Bitwarden vault has never succeeded—despite all code being correct—because bw serve (the local REST API proxy) inherits the broken DNS state.

Session 1 — March 15–16, 2026

Issue 1: Claude Code Forced Re-Authentication

Impact: High — Repeated session interruption

During the first session, Claude Code forced re-authentication at least four times. Each event interrupted active tool execution (mid-command, mid-file-write, mid-test-run), required the user to manually run /login, and consumed plan quota on re-establishing context.

The re-authentication events appeared to occur on a timer unrelated to actual session validity. The user's internet connection was active throughout (confirmed by successful ping to 8.8.8.8 during every interruption). There was no user action that triggered the logout—it happened silently during tool execution.

Observed Pattern

EventWhat Was InterruptedUser Impact
Re-auth #1File write (iCloud source module)Had to re-invoke, repeat context
Re-auth #2Test executionLost test output, had to restart
Re-auth #3Dependency installation attemptpip install killed mid-download
Re-auth #4Test executionHad to rebuild test runner approach

Recommendation for Anthropic

Re-authentication should not interrupt active tool execution. If a token refresh is needed, it should happen transparently in the background, or at minimum, wait for the current tool call to complete before prompting. Interrupting a pip install or file write mid-operation creates data corruption risk and wastes the user's plan quota on recovery.

Issue 2: macOS DNS Resolver Fragmentation

Impact: Critical — Blocked all package installation and API calls

The macOS system DNS resolver (mDNSResponder) entered a state where different programs saw different DNS resolution results. This is the central technical failure that blocked progress across both sessions.

The Fragmented State

ToolDNS Working?Notes

ping google.comNocannot resolve google.com: Unknown host

Root Cause Analysis

The initial state of the Wi-Fi interface had no DNS servers configured:

$ networksetup -getdnsservers Wi-Fi
There aren't any DNS Servers set on Wi-Fi.

The system was relying on the router-provided DNS at 192.168.1.254 (an AT&T gateway), which was either failing or returning stale results. Setting explicit DNS servers appeared to fix the issue for some tools but not others: 

$ networksetup -setdnsservers Wi-Fi 8.8.8.8 1.1.1.1
# dig: works immediately
# ping google.com: works immediately
# Python socket.getaddrinfo(): still fails
# Node.js DNS: still fails

The mDNSResponder cache appeared to be serving stale negative results to processes that use the system resolver (Python, Node, curl), while tools that implement their own resolution (dig, nslookup) worked correctly.

Key Observation: The Problem Is Process-Scoped

Critical Finding

The DNS failure appears to be inherited by child processes of Claude Code. When the user runs bw login --apikey and bw create item from their own terminal, DNS works perfectly and items are created in Bitwarden. When Claude Code runs the exact same commands, or starts bw serve, DNS fails with getaddrinfo ENOTFOUND.

This strongly suggests that the Claude Code process itself (or its sandbox environment) has a corrupted or restricted DNS resolver state that is inherited by all child processes, regardless of system-level DNS configuration.

Workarounds Attempted (Session 1)

Attempt 1

Set DNS servers via networksetup -setdnsservers Wi-Fi 8.8.8.8 1.1.1.1. Fixed dig and ping but not Python/Node resolvers.

Attempt 2

Tried sudo killall -HUP mDNSResponder to flush DNS cache. Failed: no sudo access in Claude Code sandbox.

Attempt 3

Toggled DNS off and back on: networksetup -setdnsservers Wi-Fi "Empty" then re-set. Python's resolver briefly started working. pip install -e ".[dev]" succeeded during this window. Node's resolver never recovered.

Attempt 4

Monkey-patched Python's socket.getaddrinfo() to shell out to dig for resolution. This allowed httpx to reach the Bitwarden API. Successfully authenticated via OAuth2.

Attempt 5

npm install -g @bitwarden/cli: Node.js DNS never recovered, so npm could not reach the registry. No monkey-patch equivalent available for Node.

Workarounds Attempted (Session 2)

Attempt 6

User ran sudo dscacheutil -flushcache && sudo killall -HUP mDNSResponder from their own terminal. No effect on Claude Code's resolver. Python socket.getaddrinfo('api.bitwarden.com', 443) still returned gaierror: [Errno 8] nodename nor servname provided, or not known.

Attempt 7

User ran bwsync sync --source chrome from their own terminal (where DNS works). The CLI ran, extracted 634 passwords, identified 549 unique entries. However, bw serve had been started from Claude Code's shell, so it also had broken DNS. Result: 527 of 549 push attempts failed with getaddrinfo ENOTFOUND api.bitwarden.com.

Attempt 8

Suggested user kill and restart bw serve from their terminal (so it would inherit working DNS). Session ended before this could be tested.

Issue 3: Bitwarden API Requires Client-Side Encryption

Impact: Resolved — bw CLI handles encryption transparently

During Session 1, before the bw CLI was available, we attempted to create vault items via the Bitwarden REST API directly. This revealed that the API is a zero-knowledge architecture: all cipher field values must be encrypted client-side before submission. This was ultimately resolved by installing the bw CLI (done by the user manually via brew install bitwarden-cli), which handles encryption transparently.

What Worked (Session 1, via monkey-patched DNS)

# OAuth2 token exchange: SUCCESS (HTTP 200)
POST https://identity.bitwarden.com/connect/token
  grant_type=client_credentials
  client_id=user.a0673a03-...
  scope=api
  deviceType=7
  deviceName=bwsync

# Profile retrieval: SUCCESS (HTTP 200)
GET https://api.bitwarden.com/accounts/profile
  → email: confirmed
  → encrypted symmetric key: present
  → organization membership: confirmed

What Failed (Session 1)

# Cipher creation: REJECTED (HTTP 400)
POST https://api.bitwarden.com/ciphers
{
  "type": 1,
  "name": "bwsync test entry",      ← server rejects: "not a valid encrypted string"
  "login": {
    "username": "test-user",          ← same error
    "password": "test-password"       ← same error
  }
}

Resolution (Session 2)

# bw CLI handles encryption transparently:
$ export BW_SESSION="..."
$ echo '{"type":1,"name":"test","login":{...}}' | bw encode | bw create item
# SUCCESS — item created in vault with full client-side encryption

Three test items were successfully pushed to Bitwarden this way, confirming that the API credentials are valid and the vault is accessible. The remaining blocker is purely the DNS issue preventing bw serve from reaching the API when started from Claude Code.

Issue 4: Python 3.14 Build Backend Compatibility

Impact: Medium — Resolved after two attempts

During the brief window when pip had DNS access, pip install -e ".[dev]" failed twice before succeeding on the third attempt:

Failure 1: Invalid build backend

build-backend = "setuptools.backends._legacy:_Backend"
# ERROR: Cannot import 'setuptools.backends._legacy'

This backend path does not exist in current setuptools. Fixed by changing to "setuptools.build_meta".

Failure 2: Package discovery

# ERROR: Multiple top-level packages discovered in a flat-layout

Setuptools could not auto-discover the bwsync package alongside scripts/, tests/, and hooks/. Fixed by adding explicit package discovery: 

[tool.setuptools.packages.find]
include = ["bwsync*"]

These are minor issues, but in a session where the DNS window was fleeting, each failed pip install attempt risked losing network access before the fix could be retried.

The Remaining Path to Success

What needs to happen

Everything works except for one thing: bw serve must be started from a shell that has working DNS resolution. The fix is exactly one command sequence: 

# From the user's terminal (not Claude Code):
pkill -f "bw serve"
export BW_SESSION="$(bw unlock --raw)"
bw serve --port 8087 &
bwsync sync --source chrome

This has never been tested because the DNS issue was only identified as process-scoped in the final minutes of Session 2. All application code is complete and tested. The engine extracts, deduplicates, classifies, and pushes correctly. Only the network layer between bw serve and api.bitwarden.com is broken, and only when bw serve inherits Claude Code's resolver state.

Recommendations

For Anthropic (Claude Code)

  1. Investigate the process-scoped DNS failure. The fact that the user's terminal can resolve DNS while Claude Code's child processes cannot—on the same machine, at the same time, with the same DNS configuration—suggests that Claude Code's sandbox or process environment is interfering with DNS resolution. This may be a macOS sandbox entitlement issue, a nsswitch.conf override, or a stale resolver file descriptor inherited from the parent process.
  2. Do not interrupt active tool execution for re-authentication. Queue the re-auth prompt for after the current tool call completes, or refresh tokens transparently in the background.
  3. Grant limited sudo for system maintenance. Operations like sudo killall -HUP mDNSResponder (DNS cache flush) are non-destructive and would have immediately resolved the blocking issue if the problem were system-wide. (In this case it appears process-scoped, so sudo may not help, but it should be available as a diagnostic tool.)
  4. Detect DNS failures earlier. When pip install fails with a connection error, Claude Code should run a diagnostic (dig vs getaddrinfo) and surface the discrepancy to the user rather than retrying the same failing command.
  5. Preserve session state across re-auth. Each forced logout discards the running command's state, forcing the agent to re-read files and rebuild context, wasting plan quota.

For Bitwarden

  1. Provide a Python SDK that handles encryption. The existing bitwarden-sdk package on PyPI exists but could not be installed during Session 1. A pure-Python fallback (no native extensions) would reduce the dependency on the Node.js-based CLI.
  2. Document the encryption requirement prominently in API docs. The REST API documentation does not make it immediately clear that all cipher fields must be pre-encrypted. A developer following the API reference will construct a valid-looking JSON payload, send it, and receive a cryptic "not a valid encrypted string" error with no guidance on how to encrypt the data.
  3. Consider a server-side encryption option for API key auth. When authenticating via API key (not master password), the server already has the encrypted symmetric key. An opt-in endpoint that accepts plaintext and encrypts server-side would enable automation use cases without compromising the zero-knowledge model for interactive users.
