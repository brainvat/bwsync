The problem

Your passwords are everywhere
and nowhere.

If you use Chrome, you probably have multiple profiles — one for work, one personal, maybe a few more you barely remember. Each profile has its own vault, completely isolated. Chrome has no way to consolidate them.

Then there's iCloud Keychain for Safari logins, and Google Password Manager which stores yet another set — with no programmatic API and aggressive lock-in by design.

bwsync fixes this in one run. Extract everything locally, deduplicate across sources, push to Bitwarden, keep it in sync. You'll never wonder which profile has the right password again.

⚠ Chrome Profile 7 417 passwords ⚠ Chrome Profile 1 60 passwords ⚠ Chrome Profile 11 59 passwords ○ iCloud Keychain 88 passwords ○ Google Password Mgr unknown ─────────────────────────────── ▶ Bitwarden (after) 0 → 601

Scans every Chrome profile simultaneously. Reads encrypted Login Data via the macOS Keychain. Handles discovery and deduplication automatically.

Reads directly from macOS Keychain via the native security framework. No third-party dependencies, no cloud round-trips.

⚠️

When a source password differs from Bitwarden, it's flagged for your review. Nothing is silently overwritten. Ever.

Every sync run is logged: what was pushed, what conflicted, what was skipped. Export to CSV anytime.

  • 1

    Extract

    All sources run in parallel. Chrome profiles are decrypted via macOS Keychain. iCloud Keychain is read natively. GPM CSVs are ingested from file.

  • 2

    Normalize & deduplicate

    All entries map to a common schema. Duplicates across sources are collapsed — newest used timestamp wins.

  • 3

    Diff against Bitwarden

    Each entry is compared against the live vault via bw serve. New entries are queued. Conflicts are flagged and held.

  • 4

    Push & log

    Queued entries are pushed via local REST API. Every run is written to the audit log — what changed, when, and why.

Security model

Paranoid by design.

Credentials are treated as radioactive — handled briefly, carefully, and never left lying around.

🧠

Memory only

Passwords exist in process memory during the extraction window only. Never written to the state store or any log.

#

Hashed state store

The local SQLite state store holds SHA-256 hashes of passwords for change detection — not the passwords themselves.

🗑️

Temp file hygiene

Chrome's SQLite databases are copied to tempfile.mkstemp() paths and deleted in finally blocks.

🔑

Session token via env

Bitwarden's session token is passed via BW_SESSION — never written to disk or config files.

🔒

chmod 600

The state database and all output files are created with owner-only permissions. Readable by you, invisible to other system users.

🪝

Repo protection hooks

Committed git hooks block any accidental staging or pushing of credential files. See the Protection section below.

Protection setup

Stop credentials leaking
into your repo.

This project works with sensitive data. One bad git add and your passwords are on GitHub forever. Run this once after cloning to protect yourself.

🛡️

One command sets everything up

After cloning the repo, run the setup script. It installs all protection layers, verifies they work, and tells you exactly what's covered. Re-run it after pulling if hooks have been updated.

chmod +x setup-protections.sh && ./setup-protections.sh
Layer 1 — First line of defense

.gitignore

The never-push-passwords/ directory is permanently excluded from git. Files placed there are completely invisible — git won't see them, stage them, or mention them. This is where your raw CSV exports should live temporarily.

gitignore
Layer 2 — Before commits

pre-commit hook

Runs before every git commit. Blocks any data file (.csv .json .txt .db) with a credential-sounding name. Code files like .py are never blocked — even if they say "passwords" in their name.

pre-commit
Layer 3 — Before pushing

pre-push hook

Runs before every git push. Scans the full set of commits being sent — not just the staging area. The safety net that catches anything that somehow slipped through layers 1 and 2.

pre-push

The code vs. data rule

The hooks are designed around one key distinction: a .py file named extract_chrome_passwords.py is code — it contains logic, not credentials. A .csv file named chrome_passwords_export.csv is data — it contains your actual passwords.

FileBlocked?Reason
extract_chrome_passwords.py✅ AllowedCode file — contains logic, not credentials
chrome_passwords_20260315.csv🚫 BlockedData file with credential name
never-push-passwords/export.csv🚫 BlockedInside protected directory
bwsync/sources/icloud.py✅ AllowedCode file — always safe
icloud_keychain_backup.json🚫 BlockedData file with credential name

Hooks live in hooks/ at the repo root and are version-controlled. setup-protections.sh runs git config core.hooksPath hooks/ to activate them — so updates distribute automatically when you pull, and every clone is protected from day one.

Get started

Up and running
in two minutes.

Requires Python 3.11+, the Bitwarden CLI, and macOS for Chrome + iCloud sources.

Install
$ brew install bitwarden-cli && bw login
$ git clone https://github.com/brainvat/bwsync.git && cd bwsync
$ python -m venv venv && source venv/bin/activate
$ pip install -e .
 
# Install credential protection (run once after cloning)
$ chmod +x setup-protections.sh && ./setup-protections.sh
 
# Preview what would sync — nothing is pushed
$ bwsync sync --dry-run
 
# Launch the TUI dashboard
$ bwsync
Contributing

Add a source.
Give back to the community.

bwsync is MIT licensed and built to be extended. The most impactful contribution is a new source plugin — Firefox, Dashlane, LastPass, Keeper, and others are all natural fits.

Each source is a single Python file implementing one interface. If it can read credentials and return a list of NormalizedEntry objects, it works.

Never include real credentials in test fixtures. Use the anonymized samples in tests/fixtures/. Read SECURITY.md before opening a PR.
